# Volume 07 — Project Management

**Version:** v1.0 &nbsp;|&nbsp; **Status:** Draft &nbsp;|&nbsp; **Owner:** WES &nbsp;|&nbsp; **Project:** WES Blueprint

The project execution framework for WORLD Engineering Studio (WES) — how projects are initiated, planned, run, and closed. Roles are defined in [Volume 03 — Roles](../Volume-03-Roles/README.md); the delivery workflow in [Volume 04 — Engineering System](../Volume-04-Engineering-System/README.md).

---

## Project Lifecycle

```
Initiate → Plan → Execute → Monitor → Close
```

Each phase has a clear owner (the Project Manager coordinates) and an exit condition before the next begins.

## Project Types

| Type | Description |
|------|-------------|
| **Product** | Build a full software product (e.g., Project-001: WORLD). |
| **Feature** | A defined addition to an existing product. |
| **Internal** | Studio tooling, automation, or Blueprint work. |
| **Research** | Time-boxed exploration to reduce uncertainty. |

## Project Initiation

- Define the goal, scope, and success criteria.
- Assign a Project Manager and the required roles.
- Create the project repository and baseline docs.
- Confirm the project meets the Definition of Ready ([Volume 04](../Volume-04-Engineering-System/README.md)).

## Planning Process

- Break the goal into milestones and tasks.
- Estimate effort and sequence by dependency.
- Set priorities and a target timeline.
- Identify risks and required resources up front.

## Task Breakdown

- Decompose work into small, independent tasks.
- Each task has an owner, acceptance criteria, and a linked issue.
- Tasks should be completable in one focused effort.

## Sprint Management

- Work runs in short, fixed iterations (sprints).
- Each sprint has a goal and a committed task set.
- Progress is reviewed at the end; incomplete work is re-planned.

## Milestones

- Milestones mark meaningful, verifiable progress.
- Each has a definition of completion and a target date.
- Milestone completion triggers review and stakeholder update.

## Risk Management

- Identify risks early and record them in the project docs.
- Rate by likelihood and impact.
- Assign an owner and a mitigation for each significant risk.
- Revisit risks at each milestone.

## Change Management

- Changes to scope are requested explicitly and assessed for impact.
- Significant changes require Studio Director (and, if strategic, Founder) approval per the [AI Decision Hierarchy](../Volume-05-AI-System/README.md).
- Approved changes update the plan, tasks, and docs.

## Project Closure

- Confirm all milestones and the Definition of Done are met.
- Capture lessons learned (see [Volume 09 — Knowledge Management](../Volume-09-Knowledge-Management/README.md)).
- Archive documentation and hand over or release.
